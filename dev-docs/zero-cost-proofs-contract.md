# Zero-cost proofs — implementation contract

> Audience: maintainers of the proof DSL (`dsl/web.rkt`, `dsl/private/check-runtime.rkt`).
> Originally the ZC-HARNESS Wave-0 spec; **updated to reflect the as-built system.**
>
> **AS-BUILT (status: shipped, default-on).** The erasure is implemented and is
> now the DEFAULT. A full differential audit (`DIFFPROOFS_FORCE_DSL_PURGE=1
> scripts/differential-proofs.sh`) shows 80/80 behavioral parity with the runtime
> safety net, the emitted `.rkt` is byte-identical regardless of the setting, and
> the benchmark shows ~81% less time / ~47% less allocation on proof-annotated
> calls (0 alloc on proof-free params). The sections below are the spec the
> erasure satisfies; the **default** and **Contract 1** notes have been updated
> from "future plan" to "as built".

TESL attaches GDP (Ghosts of Departed Proofs) proofs to values. Today a runtime
"safety net" wraps every proof-carrying value in a `named-value` struct and
re-validates it at every call boundary. **All of that cost lives in the Racket
DSL macro expansion, not in the OCaml emitter** — the emitter emits only macro
*calls*, and `raw-value`/`facts-of` are idempotent on raw values. Therefore the
erasure is DSL-only and the emitted `.rkt` is **byte-identical** with the toggle
on or off. The harness asserts that byte-identity as a hard invariant.

All line numbers below are against the clean-`main` baseline this worktree
branched from; re-`grep` the named identifier before editing (they drift).

---

## 1. The toggle: `TESL_ZERO_COST_PROOFS`

* **Env var name:** `TESL_ZERO_COST_PROOFS`.
* **Default: ON (erase).** Unset ⇒ erase. The net is requested explicitly with a
  falsy value: `0`, `false`, `no`, `off` (case-insensitive). Truthy values
  (`1`, `true`, `yes`, `on`) also force erase.
* **Baked at DSL-compile time.** The default is a `begin-for-syntax` constant
  (`zero-cost-proofs-default?`) evaluated when the DSL is compiled, so a tool
  switching modes between runs must purge the DSL `compiled/` (see the
  bytecode-purge requirement below). The shipped DSL is compiled with the var
  unset ⇒ erase by default.
* **Unconditional** — erases in release and `--debug` alike. (Contract 1 below was
  *removed*: a sound checker makes the runtime structs redundant; the debugger
  reads proof/type from compile-time info.)

### Where to read it — expansion time, not run time

The cost sites are generated by **macros** (`build-executable-expansion`,
`build-check-like-expansion`, `wrap-runtime-named-binding`). The gate must be a
**phase-1 / expansion-time** decision so the erased path generates *less code*,
not code that branches at run time. Read the env var in the transformer
environment (e.g. a `begin-for-syntax` constant or a helper used inside the
`syntax`/`quasisyntax` template), so the emitted, fully-expanded program for an
ON build literally lacks the wrapping/validate/parameterize forms.

```racket
;; as built (dsl/web.rkt + dsl/private/check-runtime.rkt, duplicated)
(begin-for-syntax
  (define zero-cost-proofs?            ; default-on; =0/false/no/off restores the net
    (let ([v (getenv "TESL_ZERO_COST_PROOFS")])
      (cond [(not v) #t]
            [(member (string-downcase v) '("0" "false" "no" "off")) #f]
            [else #t]))))
```

### Contract 1 — REMOVED: erasure is unconditional

Earlier drafts forced erasure OFF under `--debug` (to keep `named-value` structs
for the debugger). **That carve-out was removed.** For a sound static checker the
runtime proof structs are redundant: a binding's proof is type-level information
known at compile time (exactly what `--type-at` / hover report). The debugger
therefore needs no runtime structs — its Variables panel already shows the raw
value (`*x`), and proof/type is overlaid from compile-time type info. Breakpoint
checkpoints (`thsl-src!`) are emitted independently by the OCaml emitter, so
stepping/inspection is unaffected by erasure.

So the gate is simply `zero-cost-proofs?` (the default-on constant above), applied
in release and `--debug` alike. `TESL_ZERO_COST_PROOFS=0` restores the net for
regression comparison only. Verified: a `--debug` build emits 0
`validate-runtime-argument` (erased) yet retains its `thsl-src!` checkpoints.

> The return-value re-check (`zero-cost-proofs?/rt` in `dsl/web.rkt`) follows the
> same default uniformly (it is a no-op for statically-valid programs).

### Bytecode-purge requirement (read this or the toggle will look broken)

The env read is frozen into Racket bytecode the moment a `.zo` exists for the
module that reads it. Racket caches expanded modules in `compiled/` dirs and
does **not** re-expand a stale `.zo` just because an env var changed. Any tool
that switches the toggle between runs (the harness does) **must purge every
`compiled/` dir first**:

```sh
find . -type d -name compiled -prune -exec rm -rf {} +
```

Crucially, emitted `.rkt` files `(require tesl/dsl/…)` resolve via a global
`raco pkg --link` to the installed `tesl` checkout, so the DSL bytecode that
must be purged lives under **that** package root — not necessarily the cwd.
`scripts/differential-proofs.sh` computes the package root dynamically
(`collection-file-path "web.rkt" "tesl" "dsl"`) so it can purge it; reuse that
logic.

**Performance / correctness trade-off the harness encodes (important for you):**
purging the *shared* package-root DSL `compiled/` (i) disturbs every other tool
on the machine and (ii) forces a multi-minute full-DSL re-expansion (the DSL is
large and, in a fresh checkout, ships no `.zo` — every Racket run already pays a
~10–13s from-source DSL load; this is the same cost `compile-examples.sh`'s test
sweep pays). So the harness purges the DSL package root **only when**
`DIFFPROOFS_FORCE_DSL_PURGE=1`. It is OFF by default because **today the toggle
is a no-op**: the cached DSL expansion is identical for net-off and net-on, so
reusing it across modes is correct and fast. **The instant you wire
`TESL_ZERO_COST_PROOFS` into the DSL macros, that assumption breaks** — the two
modes then expand differently, and the harness MUST purge the DSL between them
or net-on will silently reuse net-off's expansion. Therefore:

> **ZC-SWITCH / ZC-FINALIZE: once the toggle is real, always run the audit with
> `DIFFPROOFS_FORCE_DSL_PURGE=1` (e.g. `DIFFPROOFS_FORCE_DSL_PURGE=1 make
> diff-proofs-full`).** This is the strict, slow, fully-correct mode and the gate
> for net deletion. Warm the DSL bytecode once first (run `compile-examples.sh`,
> or `raco make` the `dsl/` entry points) so the per-mode recompile is the only
> big cost.

## 2. Cost sites to ERASE (release, non-debug)

These are the allocations and dynamic-extent installs the erasure removes. The
benchmark (`tests/bench/proof-overhead.rkt`) drives exactly these primitives and
today measures (3-proof-param hot path, this machine, indicative):

| mode | ns/call | bytes/call |
|---|---:|---:|
| net-off (proof-free params) | ~10 | **0** |
| net-off (proof-annotated, erased) | ~2,900 | ~337 |
| net-on (today's safety net) | ~13,400 | ~626 |

The win to land: proof-free params → **0 alloc**; proof-annotated → **≤1 alloc**.

### 2a. Parameter wrapping via `runtime-bind+evidence`

* `dsl/private/check-runtime.rkt:136` — `(define (runtime-bind+evidence public-name value) …)`
  → allocates a `named-value` (via `runtime-evidence`) **and** a `runtime-binding`.
* Emitted by:
  * `dsl/web.rkt` `build-executable-expansion` (~`500`–`630`), at the param
    boundary: `(let-values ([(arg-evidence arg-binding) (runtime-bind+evidence quoted-name arg-id)] …) …)`.
  * `dsl/private/check-runtime.rkt` `build-check-like-expansion` (`1257`–`1358`),
    same shape for checker/auther args (~`1334`).
  * `dsl/private/check-runtime.rkt` `wrap-runtime-named-binding` (`1019`), for
    `let`-bound proof-carrying locals (~calls `runtime-bind+evidence` at `1030`).

**Erase:** bind `*arg` / `arg` directly to the **raw** value. For a proof-FREE
param, that is the entire binding — zero allocation. Do not call
`runtime-bind+evidence` for these.

### 2b. `validate-runtime-argument`

* `dsl/private/check-runtime.rkt:747` —
  `(define (validate-runtime-argument who subject public-name value expected-type [expected-proof #f] [name-env #f] [extra-bindings (hash)]) …)`
  → re-derives evidence, runs `runtime-type-satisfied?`, calls `facts-of`,
  builds an effective name-env, merges bindings, re-checks the proof.
* Emitted by `build-executable-expansion` (~`604`) and `build-check-like-expansion`
  (~`1342`), once per argument.

**Erase:** drop these calls entirely in release. The static checker
(`validation_proof.ml`, owned by ZC-FINALIZE) is the sole guarantor that the
proof held; any case where erasure changes behavior is a **static-checker gap**
the harness will surface as a behavioral diff (FAIL), to be fixed in
`validation_proof.ml`, never papered over by keeping the runtime check.

### 2c. The six-way `parameterize`

* `dsl/private/check-runtime.rkt:1343`–`1355` (checker/auther path) and the
  analogous block in `dsl/web.rkt` `build-executable-expansion` (~`605`–`615`).
* Installs, per call:
  `current-name-env`, `current-proof-env`, `current-evidence-env`,
  `current-type-env`, plus (check path) `current-check-default-value` and
  `current-check-input-facts` — a six-way `parameterize` whose right-hand sides
  call `extend-name-env` / `extend-proof-env` / `extend-evidence-env` /
  `extend-type-env`, each allocating a fresh hash.

**Erase:** remove the `parameterize` and its `extend-*-env` right-hand sides in
release. (These envs only exist to service the runtime re-validation in 2b and
the runtime name lookups in `raw-value`/`facts-of`'s extended versions; with 2b
gone they are dead.)

### 2d. Proof-annotated params keep **one** allocation — `tesl-establish-param-proof`

* `dsl/private/check-runtime.rkt:657` —
  `(define (tesl-establish-param-proof subject raw-val proof-fact) (named-value subject raw-val (list proof-fact) (hash subject raw-val)))`
  → exactly one `named-value` allocation, no validation, no `parameterize`.
* The emitter **already** emits this for proof-annotated params:
  `compiler/lib/emit_racket.ml:2214`–`2215` emits
  `(let ([n (tesl-establish-param-proof n *n `(Proof …))]) body)`.

**Keep:** for params that carry a proof annotation, bind via
`tesl-establish-param-proof` so `detachFact` / proof decomposition still resolve
the fact and bindings. This is the ≤1-alloc path. Because the emitter already
emits this form unconditionally, this is **why the `.rkt` is byte-identical**:
the erasure only changes how the *macro* treats the surrounding wrap/validate/
parameterize, not the emitted text.

### 2e. Return re-checks

* `dsl/web.rkt:1658` `validate-signature-return`, `dsl/web.rkt:1678`
  `validate-handler-return`. Gate the return re-validation in release the same
  way (static checker guarantees the returned proof). Keep the return *value*
  shape identical (raw value, or `tesl-establish-param-proof`-style single
  alloc if the return is proof-annotated and downstream decomposes it).

## 3. Carve-outs — MUST keep representation ("(almost) zero-cost")

These are **not** pure boundary re-validation; they carry or transform proof
representation that later code structurally depends on. Erasing them would
change observable behavior (the harness would FAIL them, correctly). Keep them
exactly as they are.

| Carve-out | Location | Why it stays |
|---|---|---|
| `detach` / `detach-all` / `attach` / `forget` | `dsl/private/check-runtime.rkt:1373` (`detach-proof`), `1383` (`detach-all-proof`), `1404` (`attach-proof`); `forget`/`forget-proof` (`dsl/private/evidence.rkt`) | Move/erase a fact between values; the resulting `detached-proof` / re-attached `named-value` is consumed structurally. |
| `establish` / `Fact` (trusted proofs) | `accept`/`trusted` paths feeding `named-value` | Introduce a fact with no checker; the `named-value` is the only carrier of that trust. |
| existential `pack` / `unpack` + Skolem guard | `dsl/private/check-runtime.rkt:340` (`pack/runtime`), `517` (`ensure-no-skolem-escape`) | `packed-exists` / `packed-witness` carry the hidden witness; `ensure-no-skolem-escape` enforces a real runtime invariant (Skolem escape is a runtime-only check today — see §7.9 of the spec; pair with a `KNOWN-GAP` static test). |
| `check-ok` / `check-fail` control flow | `dsl/private/evidence.rkt:34`–`35`; `check-fail?` short-circuits in `wrap-runtime-named-binding` (~`1019`) | These are the result ADT of `check`; they are control flow, not a proof safety net. Erasing them breaks `if`/`case` over check results. |
| `newtype-value` | `dsl/types.rkt:261` | Survives **automatically** — `raw-value` never unwraps a `newtype-value` (it only unwraps `named-value`/`check-ok`). No action needed, but do not add unwrapping. |
| SQL `FromDb` proofs | `dsl/sql.rkt:516`, `546` | DB-origin facts (`(FromDb …)`) are produced by the SQL layer and consumed by handlers; they are data, not a re-validation wrapper. |

`raw-value` (`dsl/private/evidence.rkt:49`) and `facts-of`
(`dsl/private/evidence.rkt:66`) are **idempotent on raw values** (a plain value
→ itself / `'()`), which is what makes erasure safe: downstream `(raw-value x)`
calls in already-emitted code keep working whether `x` is wrapped or raw.

> **`current-in-forall-filter` (already a partial erasure, leave it):**
> `dsl/private/check-runtime.rkt:764` already **skips** the runtime proof check
> for `List.filterCheck` / `List.allCheck` elements (ForAll is compile-time-only
> there; elements carry no runtime structs). The runtime net is already OFF on
> that path — a known gap-hunter (plan: "ForAll element leak") that must be a
> static rejection. Do not "fix" it by re-enabling a runtime check.

## 4. What the harness verifies (so you know what "done" looks like)

`scripts/differential-proofs.sh` (run via `make diff-proofs`, full corpus via
`make diff-proofs-full`):

1. **Byte-identity of emission.** Compiles each testable `.tesl` with the env
   unset and with `TESL_ZERO_COST_PROOFS=1`; asserts the produced `.rkt` are
   `cmp`-identical. **If you change the emitter, this FAILS** — the toggle leaked
   out of the DSL. Keep all erasure in the DSL.
2. **Behavioral equivalence.** Runs each file's `(module+ test …)` submodule
   net-off then net-on, **purging `compiled/` between modes**, normalizes
   gensyms / timestamps / UUIDs / durations / pointers, and diffs. Identical
   normalized output + identical exit ⇒ PASS. **A behavioral diff ⇒ a
   static-checker gap** to fix in `validation_proof.ml` (ZC-FINALIZE), not to
   hide by retaining a runtime check.

`make test-net-off` runs the positive corpus test submodules with the net
disabled (`TESL_ZERO_COST_PROOFS=1`) and reports pass/fail (Verification step 4:
every positive `expect`/`expectHasProof` still passes with the net off).

`make bench` prints the `{net-off, net-on}` ns/call + bytes/call table. After
your change, the net-off proof-annotated row should drop to ≤1 alloc and the
proof-free row to 0; ns/call should fall sharply. `--check-threshold
--max-on-bytes N` is wired for Verification step 5 (ON-mode bytes/call under a
committed threshold) but is OFF by default until you land.

### Pre-existing corpus state observed by ZC-HARNESS (for ZC-FINALIZE)

When the harness was built (clean-`main` baseline), the corpus already had
**runtime failures that are independent of the toggle** — they fail *identically*
net-off and net-on, so `differential-proofs.sh` correctly reports them as PASS
(no divergence), but `make test-net-off` reports them as FAIL (their `test`
blocks don't pass at runtime, in any mode):

* `example/learn/lesson54-debugging-proof-errors.tesl` — genuine bug:
  `trusted-proof: unbound GDP name in proof template: (*n)` (fails net-off too).
* Much of `tests/*.tesl` is the **adversarial / critical-review** suite
  (`adversarial-review-tests`, `critical-review-*`); several of those `test`
  blocks intentionally exercise failing/edge behavior and exit non-zero, and
  their committed `.rkt` are in places stale vs the current compiler. Verified
  pre-existing: a fresh net-off batch fails the same files.

Implication: `make test-net-off` exits non-zero **today** due to this
pre-existing breakage, NOT due to anything zero-cost-related. ZC-FINALIZE owns
corpus health — when triaging post-erasure, separate (a) these pre-existing
failures (present before your change; fix or quarantine them) from (b) any NEW
behavioral diff that `differential-proofs.sh` flags after the erasure lands
(those are static-checker gaps to fix in `validation_proof.ml`). The
differential harness is the precise instrument for (b): it only flags a file
when net-on and net-off *diverge*.
