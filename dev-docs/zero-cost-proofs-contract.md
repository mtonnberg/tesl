# Zero-cost proofs (the only mode)

> Audience: maintainers of the proof DSL (`dsl/web.rkt`, `dsl/private/check-runtime.rkt`).
>
> **AS-BUILT (status: shipped, the only mode).** Proof erasure is implemented and
> is the sole behavior — there is no longer any toggle, no env var, and no runtime
> "safety net" to fall back on. The compile-time checker is the only proof /
> declared-context contract. When parity was being established, a gated env var
> and a differential audit oracle compared "net off" vs "net on"; once 80/80
> behavioral parity and byte-identical emission were confirmed, the gate and the
> differential audit harness were **removed** — the two modes can no longer differ,
> so the comparison is meaningless. This document records WHAT is erased and the
> carve-outs that remain.

TESL attaches GDP (Ghosts of Departed Proofs) proofs to values. **All of the
proof-tracking cost lives in the Racket DSL macro expansion, not in the OCaml
emitter** — the emitter emits only macro *calls*, and `raw-value`/`facts-of` are
idempotent on raw values. The erasure is therefore DSL-only: for a proof-FREE
binding the emitted `.rkt` is identical to (and as cheap as) the unannotated
form, because what changes is how the *macros* expand, not the emitted text. A
proof-ANNOTATED parameter is NOT byte-identical — the emitter writes the
annotation into the macro header (`[x : Integer ::: (InRange x)]`) and the
expansion binds it via `tesl-establish-param-proof`, retaining one `named-value`
carrier (the ≤1-alloc carve-out enumerated in §3). The "byte-identical" property
holds for proof-free bindings, not for proof-carrying ones.

"Zero-cost" refers specifically to *proof erasure*: proof TRACKING is erased (no
proof structs on the standard path), not to all runtime overhead. Each
fn/handler call still pays a small always-on capability-grant + return-shape
validation cost — see §4 for the full list of always-on checks that are NOT
erased.

All line numbers below are indicative against an earlier baseline; re-`grep` the
named identifier before editing (they drift).

---

## 1. What erasure means

* **Erasure is unconditional.** It applies in release and `--debug` alike. A sound
  static checker makes the runtime proof structs redundant: a binding's proof is
  type-level information known at compile time (exactly what `--type-at` / hover
  report). The debugger therefore needs no runtime structs — its Variables panel
  shows the raw value (`*x`), and proof/type is overlaid from compile-time type
  info. Breakpoint checkpoints (`thsl-src!`) are emitted independently by the
  OCaml emitter, so stepping/inspection is unaffected by erasure.
* **The static checker is the sole contract.** Any case where erasing the runtime
  structs would change behavior is a **static-checker gap** — a compiler bug to
  fix in `validation_proof.ml`, never something a runtime check papers over.
  There is no runtime mode that re-installs the old structs.
* **Expansion time, not run time.** The cost sites are generated by macros
  (`build-executable-expansion`, `build-check-like-expansion`,
  `wrap-runtime-named-binding`). Erasure is a phase-1 / expansion-time decision so
  the fully-expanded program literally lacks the wrapping/validate/parameterize
  forms — not code that branches at run time.

---

## 2. Cost sites that are ERASED

These are the allocations and dynamic-extent installs that erasure removes. The
benchmark (`tests/bench/proof-overhead.rkt`) drives exactly these primitives and
measures (3-proof-param hot path, indicative): proof-free params → **0 alloc**;
proof-annotated → **≤1 alloc**; ~81% less time / ~47% less allocation than the
old safety net.

### 2a. Parameter wrapping via `runtime-bind+evidence`

* `dsl/private/check-runtime.rkt` — `(define (runtime-bind+evidence public-name value) …)`
  → allocates a `named-value` (via `runtime-evidence`) **and** a `runtime-binding`.
* Emitted at the param boundary by `build-executable-expansion` (`dsl/web.rkt`),
  by `build-check-like-expansion` (checker/auther args), and by
  `wrap-runtime-named-binding` (`let`-bound proof-carrying locals).

**Erased:** bind `*arg` / `arg` directly to the **raw** value. For a proof-FREE
param, that is the entire binding — zero allocation; `runtime-bind+evidence` is
not called.

### 2b. `validate-runtime-argument`

* `dsl/private/check-runtime.rkt` —
  `(define (validate-runtime-argument who subject public-name value expected-type …) …)`
  → re-derives evidence, runs `runtime-type-satisfied?`, calls `facts-of`,
  builds an effective name-env, merges bindings, re-checks the proof.
* Emitted once per argument by `build-executable-expansion` and
  `build-check-like-expansion`.

**Erased:** dropped entirely. The static checker (`validation_proof.ml`) is the
sole guarantor that the proof held.

### 2c. The six-way `parameterize`

* The checker/auther path and the analogous block in `build-executable-expansion`
  install, per call: `current-name-env`, `current-proof-env`,
  `current-evidence-env`, `current-type-env`, plus (check path)
  `current-check-default-value` and `current-check-input-facts` — each right-hand
  side calling an `extend-*-env` that allocates a fresh hash.

**Erased:** the `parameterize` and its `extend-*-env` right-hand sides are gone.
(These envs only existed to service the runtime re-validation in 2b and the
runtime name lookups in `raw-value`/`facts-of`'s extended versions; with 2b gone
they are dead.)

### 2d. Return re-checks

* `validate-signature-return` / `validate-handler-return` (`dsl/web.rkt`). The
  return re-validation is gone (the static checker guarantees the returned proof);
  the return *value* shape is identical (raw value, or a single-alloc carrier if
  the return is proof-annotated and downstream decomposes it).

## 3. Exceptions — these RETAIN a token ("(almost) zero-cost")

These are **not** pure boundary re-validation; they carry or transform proof
representation that later code structurally depends on. Erasing them would change
observable behavior. They are kept exactly as they are.

| Exception | Location | Why it stays |
|---|---|---|
| Proof-annotated params — `tesl-establish-param-proof` | `dsl/private/check-runtime.rkt` (`tesl-establish-param-proof`); emitted by `compiler/lib/emit_racket.ml` | A param that carries a `:::` annotation binds via `tesl-establish-param-proof`, allocating exactly **one** `named-value` (no validation, no `parameterize`) so `detachFact` / proof decomposition still resolve the fact and bindings. This is the ≤1-alloc path. |
| `detach` / `detachFact` / `attach` / `attachFact` / `forget` | `dsl/private/check-runtime.rkt` (`detach-proof`, `detach-all-proof`, `attach-proof`); `dsl/private/evidence.rkt` (`forget`) | Move/erase a fact between values; the resulting `detached-proof` / re-attached `named-value` is consumed structurally — first-class proof tokens passed around at runtime. |
| `establish` / `Fact` (trusted proofs) | `accept`/`trusted` paths feeding `named-value` | Introduce a fact with no checker; the `named-value` is the only carrier of that trust. |
| existential `pack` / `unpack` + Skolem guard | `dsl/private/check-runtime.rkt` (`pack/runtime`, `ensure-no-skolem-escape`) | `packed-exists` / `packed-witness` carry the hidden witness; the witness-escape check (below) enforces a real runtime invariant. |
| `newtype-value` | `dsl/types.rkt` | Survives **automatically** — `raw-value` never unwraps a `newtype-value` (only `named-value`/`check-ok`). No action needed; do not add unwrapping. |
| SQL `FromDb` proofs | `dsl/sql.rkt` | DB-origin facts (`(FromDb …)`) are produced by the SQL layer and consumed by handlers; they are data, not a re-validation wrapper. |

`raw-value` and `facts-of` (`dsl/private/evidence.rkt`) are **idempotent on raw
values** (a plain value → itself / `'()`), which is what makes erasure safe:
downstream `(raw-value x)` calls in already-emitted code keep working whether `x`
is wrapped or raw.

## 4. Always-on runtime checks (NOT erased)

Erasure removes the *proof re-validation* net. The following runtime checks are
orthogonal and remain ON unconditionally — they guard runtime-only facts the
static checker cannot establish:

* **Ambient capability-grant.** The runtime installs the *whole-app capability
  union* as the ambient dynamic-extent (`dsl/capability.rkt`
  `with-capabilities`); there is no per-handler narrowing. Each handler's
  `call-with-declared-capabilities` is a subset-*assertion* (`declared ⊆ ambient`
  via `require-capabilities!`) — it checks, but does not narrow to, the handler's
  `requires [...]` row. Per-handler narrowing (CAP-A2) was attempted and reverted;
  it is not yet done, blocked on making per-callsite capability inference complete
  (roadmap/later). Capabilities are still granted/checked at run time, not erased
  with proofs. Note that `auth`/`check`/`establish` bodies are now *statically*
  capability-checked at compile time (they must declare the effects they use —
  `compiler/lib/validation_capabilities.ml` `cap_check_kind_info`).
* **Parameter type validation.** The runtime type check on incoming values
  (`runtime-type-satisfied?`) at the boundary stays — it validates *shape* of data
  arriving from outside the program (request bodies, DB rows), which is not a
  compile-time fact about an in-program binding.
* **Return type / shape validation.** The return *shape* check remains (distinct
  from the erased return *proof* re-check in §2d).
* **Existential-witness escape.** `ensure-no-skolem-escape` is a genuine
  runtime-only invariant (a Skolem witness must not leak out of its `unpack`
  scope); it is kept.

---

## 5. Why there is no toggle anymore

While erasure was being landed it sat behind an env var and a differential audit:
the compiler was run with the var unset vs set and the emitted `.rkt` plus the
test-submodule behavior were asserted byte-/behavior-identical. That oracle had
two outputs only because two modes existed. With the gate removed there is exactly
one mode (erase), so:

* there is no env var to read and no "restore the net for regression comparison"
  path;
* the differential audit script and the `make` targets that drove it were
  deleted — they compared identical-by-construction outputs;
* a behavioral surprise on a proof-carrying program is now diagnosed as a
  **static-checker gap** (fix in `validation_proof.ml`) or an emitter bug, not by
  flipping a runtime mode.

The benchmark (`tests/bench/proof-overhead.rkt`, `make bench`) still prints
ns/call + bytes/call for the proof-heavy hot path so regressions in the erased
path are visible.
