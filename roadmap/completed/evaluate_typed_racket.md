## Background

We currently use Racket as the runtime which is untyped. We would like our runtime to be as correct *and* as fast/memory efficient as possible and that is achievable in several ways.

One avenue to explore is to use "Typed Racked", which is possible to gradually introduce.

## Goal

To evaluate
- if it is possible to gradually introduce Typed Racket to our runtime
- if doing so would keep/improve runtime performance
- which areas would be most rewarding to type
- Is the "Optimization Coach" a good tool?
- Writing a report that either discards the idea or give concrete guidance on how to tackle this problem in a roadmap/later item.

Feasibility, Performance Implications, and Best Practices for Gradual Type Migration to Typed RacketThe integration of static type systems into dynamically typed codebases has emerged as a major paradigm in modern software engineering. Within the Racket ecosystem, this transition is facilitated by Typed Racket, a sister language designed specifically for sound gradual typing. Deciding whether to migrate an existing Racket codebase to Typed Racket, assessing its viability, and executing the transition effectively requires a deep understanding of Typed Racket’s structural design, its performance characteristics, and the engineering trade-offs involved in sound gradual typing.Feasibility and Theoretical Foundations of Gradual TypingMigrating an existing Racket codebase to Typed Racket is structurally possible and supported by the language's design, but the systemic utility of this transition depends on the application's runtime characteristics, performance requirements, and syntactic patterns. Typed Racket is structurally integrated with Racket via the #lang mechanism, which permits typed and untyped modules to co-exist and interoperate within the same application execution path. This interoperability is achieved through a contract system that monitors boundaries between typed and untyped code at run-time.From a type-theoretic perspective, sound gradual typing provides guarantees that differ fundamentally from optional type systems like TypeScript, Dart, or Python's mypy. In optional type systems, types are erased during compilation, and no runtime checks are inserted at the boundary between typed and untyped code. Consequently, optional typing does not enforce type soundness at runtime and cannot validate the $\eta$ law for different types. Conversely, sound gradual typing ensures that the static type annotations remain correct even when values from dynamically-typed code enter statically-typed fragments. This relationship is formalized using the proarrow equipment framework, where the dynamism relation $A \sqsubseteq B$ denotes a coreflection or an embedding-projection pair between the types $A$ and $B$, ensuring that refining a type to be less dynamic preserves or refines the program's underlying behavior.While the theoretical guarantees of sound gradual typing are robust, practical implementations reveal a complex landscape of benefits and challenges. Introducing static types improves code maintainability, self-documentation, and compile-time error detection. It also unlocks type-driven compiler optimizations that can dramatically accelerate certain classes of code, such as numerical computations and matrix operations, which can run 25 to 50 times faster in Typed Racket than in untyped Racket.However, the cost of enforcing soundness at runtime can be exceptionally high. When typed and untyped modules interact, Typed Racket wraps boundary-crossing values in chaperones or proxies to ensure that untyped code does not violate the invariants expected by typed code. The Gradual Typing Performance (GTP) project designed a systematic evaluation method that tests all $2^n$ configurations of $n$ modules within a migrating codebase to generate a performance lattice. The empirical findings of the GTP project indicate that intermediate, partially typed configurations of sound gradually typed codebases frequently suffer from disastrous performance degradation, with some mixed-typed configurations experiencing slowdowns of up to 35 times. This boundary-monitoring overhead means that the cost of soundness is often intolerable for production systems unless the migration is carefully planned. Additionally, even fully typed configurations remain bound to Racket's largely untyped runtime library, limiting the upper bound of type-driven optimizations.Architectural and Runtime Implications of SoundnessThe core tension in gradual typing lies in the enforcement of type boundaries. Understanding how Racket translates static type annotations into run-time checks is essential for preventing catastrophic performance regressions. For simple flat values, such as integers or strings, this check is a one-time cost. However, for higher-order values—such as functions, objects, vectors, and lists—the check cannot be performed immediately. For example, if an untyped module passes a list of integers to a typed module expecting (Listof Integer), a Deep check requires traversing the entire list to verify that every single element is an integer. If a function type (-> Integer Integer) is passed, Typed Racket cannot verify the function's behavior statically; instead, it wraps the function in a contract chaperone that intercepts every invocation to verify that the argument is an integer and that the return value is also an integer. Shallow Typed Racket replaces deep contract wrappers with first-order "shape" checks. Invariants within the typed module are then verified incrementally at run-time as elements are accessed or values are eliminated. In particular, while the Transient semantics limits boundary costs, implementing collaborative blame tracking via a global blame map incurs a massive performance penalty, leading developers to disable the Transient blame tracking in standard Shallow Racket to make the system performant. ## Tesl-specific evaluation — VERDICT: DISCARD (for the runtime)

Evaluated 2026-06-30 (core_polish) against the actual runtime (`dsl/` + `tesl/`).
The general background above is generic; this is the Tesl-specific finding.

**Verdict: DISCARD Typed Racket as a runtime performance strategy.** The Tesl
runtime is the canonical *anti-pattern* for sound gradual typing:

- **Topology funnels through one struct hub.** `dsl/private/evidence.rkt` defines
  the GDP structs (`named-value`, `check-ok`, `packed-exists`, …) that virtually
  every module consumes (`raw-value`) and produces (`attach`/`check-ok`). Few true
  leaves; none of the *interesting* perf modules (`int`/`string`/`list`/`dict`/
  `uuid`/`jwt`/`sql`/`web`) are leaves — they all import the hub.
- **Worst-case for Deep TR.** Runtime values are heterogeneous, recursive,
  higher-order (closures inside `packed-exists` bodies, codec/getter closures),
  partly polymorphic (`Any`-typed fields), and the universal type key `type-ref`
  is `#:prefab` — exactly the collection/closure/prefab shapes the general writeup
  flags as 10–35× boundary hazards and multi-minute compiles. These structs cross
  module lines on essentially every stdlib call.
- **No payoff.** The proof-free hot path is already **8.4 ns/call, 0 bytes
  allocated** (`tests/bench/proof-overhead.rkt --quick`, measured) — plain
  monomorphic Racket the CS JIT handles optimally. TR cannot beat zero allocation;
  a typed↔untyped boundary would *add* allocation.
- **Soundness lives in OCaml.** Proofs are erased by default
  (`check-runtime.rkt:894–917`: the static checker is the sole guarantor), so TR
  on the runtime adds ~nothing to the trusted surface. Neither the perf nor the
  correctness case is strong.

**Better lever: Optimization Coach** (installed, independent of TR, works on plain
`#lang racket`, zero boundary risk) — the recommended first action. If TR is
pursued at all, ONLY *Optional* typing (erased, zero runtime cost) on a single
leaf (`dsl/private/evidence.rkt`) as a documentation experiment, with hard kill
criteria (any `ns/call`/`bytes/call` regression; a hard ban on Deep/Shallow typing
`check-runtime.rkt`/`types.rkt`/`web.rkt`).

The actionable, low-risk follow-ups are captured in
`roadmap/later/optimization_coach_and_optional_typing_pilot.md`. This evaluation
item is complete; sound gradual (Deep/Shallow) typing of the Tesl runtime is
**contraindicated**.
