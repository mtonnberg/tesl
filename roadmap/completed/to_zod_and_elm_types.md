# Client Code Generation: IR + Dual Emitters

## Goal

Extend the Tesl compiler toolchain with a `tesl generate` subcommand family that produces
type-safe client code for frontend teams from a Tesl API definition.
Two targets: **TypeScript/Zod** (pragmatic, wide adoption) and **Elm** (natural semantic fit,
"best in class" showcase). Both share a common **JSON IR** emitted by the compiler.

JSON codecs (Phase 1) are a prerequisite and are already complete.

---

## What holds water from the brainstorm

- **Custom IR beats OpenAPI** as primary source of truth. OpenAPI can describe shapes but cannot
  express GDP proof semantics. Own IR retains 100% of Tesl's meaning; OpenAPI can be generated
  from the IR secondarily if needed.
- **Branded types (TS) / Opaque types (Elm)** are the correct frontends equivalents of Tesl facts.
  - TS: `string & { readonly __brand: "ValidTitle" }` with Zod `.brand<"ValidTitle">()`
  - Elm: module-opaque type + `mtonnberg/refinement-proofs`
- **`check` function AST extraction** is feasible for simple patterns (string length, int bounds,
  regex) → native Zod `.refine()` / Elm validation. Complex/opaque check functions are marked
  `"server_only"` in the IR; emitters emit a comment warning and skip inline validation.
- **ADTs map perfectly to Elm custom types** + decoders, with no impedance mismatch.
- **`FromDb` proof on responses** becomes a branded tag that frontend functions can require,
  preventing accidental use of optimistic/unsaved UI state.
- **Elm is the better "proof of concept"** for full-stack proof propagation (nominal types,
  no runtime exceptions philosophy). TypeScript/Zod is the better adoption driver.

## Phases

### Phase 1 — JSON IR (compiler change)

Add `tesl generate ir <file.tesl>` that emits a JSON file describing the compiled module.

**IR schema:**

```json
{
  "module": "todo-api",
  "records": [
    {
      "name": "Todo",
      "fields": [
        { "name": "id",    "type": "Int",    "facts": [] },
        { "name": "title", "type": "String", "facts": ["ValidTitle"] }
      ],
      "codec": "Todo"
    }
  ],
  "adts": [
    {
      "name": "JobResult",
      "variants": [
        { "tag": "Delivered", "fields": [{ "name": "messageId", "type": "String" }] },
        { "tag": "Failed",    "fields": [{ "name": "reason",    "type": "String" }] }
      ],
      "codec": "JobResult"
    }
  ],
  "newtypes": [
    { "name": "UserId", "base": "String" }
  ],
  "facts": [
    {
      "name": "ValidTitle",
      "base_type": "String",
      "logic": {
        "kind": "simple",
        "constraints": [
          { "op": "gte", "fn": "String.length", "value": 3 },
          { "op": "lte", "fn": "String.length", "value": 120 }
        ]
      }
    },
    {
      "name": "FromDb",
      "base_type": "opaque",
      "logic": { "kind": "server_only" }
    }
  ],
  "endpoints": [
    {
      "method": "GET",
      "path": "/todos/:todoId",
      "captures": [
        { "name": "todoId", "type": "String", "fact": "TodoId", "checker": "todoIdCapture" }
      ],
      "body": null,
      "response": { "type": "Todo", "fact": "FromDb", "codec": "Todo" }
    },
    {
      "method": "POST",
      "path": "/todos",
      "captures": [],
      "body": { "type": "NewTodo", "codec": "NewTodo" },
      "response": { "type": "Todo", "fact": "FromDb", "codec": "Todo" }
    }
  ]
}
```

**Compiler changes (compile_tesl.py):**

- New function `emit_ir(module) -> dict` walks the compiled module and produces the IR dict.
- New flag `--ir` in `argparse`; outputs JSON to stdout.
- `collect_simple_check_constraints(fn_form)` — detects patterns like:
  - `String.length(x) >= N` → `{"op": "gte", "fn": "String.length", "value": N}`
  - `String.length(x) <= N` → `{"op": "lte", "fn": "String.length", "value": N}`
  - `Int >= N` / `Int <= N`
  - `String.matches(x, regex)` → `{"op": "regex", "pattern": "..."}`
  - Anything else → `{ "kind": "server_only" }`
- Collect fact names from each `establish`/`check`/`auth` function.
- Collect endpoint metadata from `api` blocks.
- ~400 lines of new Python.

**CLI:**

```bash
tesl generate ir  <file.tesl>   # emit IR JSON to stdout
tesl generate ts  <file.tesl>   # emit TypeScript/Zod client
tesl generate elm <file.tesl>   # emit Elm module
```

Add `generate` subcommand to shell.nix that dispatches to the appropriate emitter script.

### Phase 2 — TypeScript / Zod Emitter

New file: `tesl/private/emit_ts.py`
Input: IR JSON (stdin or file)
Output: `<basename>-api.ts`

**Generated output for the Todo example:**

```typescript
// Generated by tesl generate ts — do not edit by hand

import { z } from "zod";

// --- Facts (branded types) ---

export type ValidTitle = string & { readonly __brand: "ValidTitle" };
export type FromDb     = { readonly __brand_FromDb: unique symbol };
export type TodoId     = string & { readonly __brand: "TodoId" };

// --- Fact validators ---

/** Tesl check: ValidTitle — server-validated rule */
export const ValidTitleSchema = z.string().min(3).max(120).brand<"ValidTitle">();

// TodoId: server-only fact — must come from a trusted API response
export const unsafe_TodoId = (val: string): TodoId => val as unknown as TodoId;

// --- Records ---

export const TodoSchema = z.object({
  id:    z.number(),
  title: ValidTitleSchema,
});
export type Todo = z.infer<typeof TodoSchema> & FromDb;

export const NewTodoSchema = z.object({
  title: ValidTitleSchema,
});
export type NewTodo = z.infer<typeof NewTodoSchema>;

// --- API client ---

const BASE = "";  // set via configure()
export function configure(base: string) { (globalThis as any).__teslBase = base; }
const base = () => (globalThis as any).__teslBase ?? BASE;

export async function getTodo(todoId: TodoId): Promise<Todo> {
  const res = await fetch(`${base()}/todos/${todoId}`);
  if (!res.ok) throw new Error(`GET /todos/:todoId failed: ${res.status}`);
  return TodoSchema.parse(await res.json()) as Todo;
}

export async function createTodo(body: NewTodo): Promise<Todo> {
  const res = await fetch(`${base()}/todos`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(NewTodoSchema.parse(body)),
  });
  if (!res.ok) throw new Error(`POST /todos failed: ${res.status}`);
  return TodoSchema.parse(await res.json()) as Todo;
}
```

**Key design decisions:**
- Response types carry `& FromDb` intersection — functions that require a DB-backed record
  will only accept values returned from the API client, not hand-constructed objects.
- `server_only` facts expose an `unsafe_Xxx` escape hatch with a name that makes the
  risk visible in code review.
- Simple fact validators (`ValidTitle`) emit native Zod `.min()/.max()/.regex()` — fully
  functional, no extra dependencies.
- Emitter is ~400 lines of Python; no external dependencies beyond `zod` on the TS side.

### Phase 3 — Elm Emitter

New file: `tesl/private/emit_elm.py`
Input: IR JSON (stdin or file)
Output: `Api/Generated.elm`

**Generated output for the Todo example:**

```elm
module Api.Generated exposing
    ( Todo, NewTodo
    , ValidTitle, validTitle
    , TodoId, todoId
    , todoDecoder, newTodoEncoder
    , getTodo, createTodo
    )

{-| Generated by tesl generate elm — do not edit by hand -}

import Http
import Json.Decode as D
import Json.Encode as E


-- Facts (opaque types — constructors not exposed)

type ValidTitle = ValidTitle String

validTitle : String -> Maybe ValidTitle
validTitle s =
    if String.length s >= 3 && String.length s <= 120 then
        Just (ValidTitle s)
    else
        Nothing

type TodoId = TodoId String

-- TodoId is server-minted; construct only from decoded API responses


-- Records

type alias Todo =
    { id    : Int
    , title : ValidTitle
    }

type alias NewTodo =
    { title : ValidTitle }


-- Decoders

todoDecoder : D.Decoder Todo
todoDecoder =
    D.map2 Todo
        (D.field "id" D.int)
        (D.field "title" (D.string |> D.andThen (\s ->
            case validTitle s of
                Just vt -> D.succeed vt
                Nothing -> D.fail "Failed Tesl proof: ValidTitle"
        )))


-- Encoders

newTodoEncoder : NewTodo -> E.Value
newTodoEncoder nt =
    let (ValidTitle rawTitle) = nt.title
    in E.object [ ( "title", E.string rawTitle ) ]


-- API helpers

getTodo : TodoId -> (Result Http.Error Todo -> msg) -> Cmd msg
getTodo (TodoId id) toMsg =
    Http.get
        { url = "/todos/" ++ id
        , expect = Http.expectJson toMsg todoDecoder
        }

createTodo : NewTodo -> (Result Http.Error Todo -> msg) -> Cmd msg
createTodo body toMsg =
    Http.post
        { url = "/todos"
        , body = Http.jsonBody (newTodoEncoder body)
        , expect = Http.expectJson toMsg todoDecoder
        }
```

**Key design decisions:**
- Facts are true Elm opaque types — the constructor is not exposed, so the rest of the
  application cannot construct them arbitrarily.
- Simple fact logic is compiled into a `Maybe`-returning smart constructor — the Elm compiler
  forces callers to handle the `Nothing` case (i.e., invalid input is a compile-time concern).
- `server_only` facts (like `FromDb` / `TodoId`) have no exported smart constructor;
  they can only be obtained via the generated decoder from a real HTTP response.
- ADT decoders use the tag/fields JSON format that Tesl already emits for JSONB columns.
- Emitter is ~500 lines of Python.

## Implementation order

1. **Phase 1 (IR)** — implement `emit_ir()` in compile_tesl.py + `--ir` flag + `tesl generate`
   shell subcommand. Write 10 Python tests covering records, ADTs, newtypes, simple/opaque facts,
   and endpoints. (~1 session)

2. **Phase 2 (TS/Zod)** — implement `emit_ts.py`, add to `tesl generate ts`. Write tests that
   snapshot-compare generated TS against known-good output for todo-api and admin-task-api.
   (~1 session)

3. **Phase 3 (Elm)** — implement `emit_elm.py`, add to `tesl generate elm`. Same snapshot approach.
   (~1 session)

**What stays out of scope (for now):**
- Complex fact logic (DB calls, third-party libs): always `server_only`, no attempt to compile.
- WebAssembly / isomorphic validation.
- OpenAPI generation from the IR.
- tRPC-style router / real-time subscription client stubs.
- Historical decoder variants in the generated Elm (Phase 3 of codecs is a separate item).
